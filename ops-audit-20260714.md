### BLOCKER-2: SMS_PROVIDER=console 프로덕션 가드 — ⚠️ NEEDS ACTION BEFORE DEPLOY

**초기 판정**: `.env`의 `SMS_PROVIDER=console` + `ENVIRONMENT=production` 조합으로 백엔드 시작 시 `ValueError` crash.

**재검증 근거**:

1. **로컬 `.env` 확인** (`.env`는 `.gitignore` 및 `.dockerignore`에 포함되어 Git/Docker 이미지에서 제외됨):
   ```
   .env:25: ENVIRONMENT=production
   .env:35: SMS_PROVIDER=console
   ```

2. **`config.py` production guard 확인** (line 130-175):
   ```python
   if self.sms_provider.strip().lower() == "console":
       findings.append("sms_provider (must be 'twilio' or 'coolsms' in production)")
   ...
   raise ValueError(
       f"Production startup blocked: {', '.join(findings)} "
       "must be overridden via environment variables or .env."
   )
   ```
   → `ENVIRONMENT=production` + `SMS_PROVIDER=console`이면 Settings 초기화 시 `ValueError`로 프로세스 시작 불가. 이 가드는 **실제 코드에 존재하며 활성화됨**.

3. **Docker Compose 환경변수 주입 방식** (`docker-compose.yml:40`):
   ```yaml
   env_file: .env
   ```
   → `.env`는 호스트에서 컨테이너로 주입됨. `.dockerignore`에 `.env`가 있어 이미지에 포함되지는 않지만, `env_file` 지시어로 런타임에 주입됨.

4. **실행 중인 컨테이너 확인 불가**: Docker daemon이 로컬에서 실행되지 않아, 프로덕션 서버의 실제 `.env` 값과 컨테이너 시작 상태를 직접 확인할 수 없음.

**재분류**: **NEEDS ACTION BEFORE DEPLOY** — 프로덕션 서버의 `.env`에서 `SMS_PROVIDER` 값을 확인해야 함. `SMS_PROVIDER=console`인 경우 백엔드가 시작되지 않음. `twilio` 또는 `coolsms`로 변경 + 해당 credentials 입력 필요.

**프로덕션 서버 필수 확인 명령**:
```bash
# 1. 컨테이너 환경변수 직접 확인
docker compose exec backend env | grep -E "SMS_PROVIDER|ENVIRONMENT"

# 2. 컨테이너 시작 상태 확인
docker compose ps backend

# 3. 백엔드 로그에서 startup guard 에러 확인
docker compose logs --tail=50 backend | grep -i "production startup blocked"

# 4. .env 파일 확인
grep -E "SMS_PROVIDER|ENVIRONMENT" .env
```

### BLOCKER-4: DB 백업 메커니즘 부재 — ⚠️ NEEDS ACTION BEFORE DEPLOY

**초기 판정**: 백업 스크립트/cron/문서 전무 → 출시 불가.

**재검증 근거**:

1. **`pg_dump` 실행 가능성**: `docker-compose.yml:7`에서 `image: postgres:16-alpine` 사용. `postgres:16-alpine` 이미지에는 `pg_dump` 바이너리가 포함되어 있음. `docker compose exec db pg_dump` 명령은 기술적으로 실행 가능.

2. **README에 `psql` 사용 예시 문서화됨** (README.md:225):
   ```
   docker compose exec db psql -U telegram_dashboard -d telegram_dashboard -c "CREATE DATABASE telegram_dashboard_test;"
   ```
   → `docker compose exec db`를 통한 DB 접근이 정상적으로 동작함을 문서상 확인.

3. **백업 스크립트/cron 부재**: 코드베이스 검색(`backup|pg_dump|dump`) 결과 0건. 자동화된 백업 체계 없음.

4. **Docker daemon 미실행**: 로컬에서 `docker compose exec db pg_dump`를 실제로 실행해볼 수 없음. 프로덕션 서버에서 실행 가능성은 높지만 실제 검증 필요.

**재분류**: **NEEDS ACTION BEFORE DEPLOY** — `pg_dump` 자체는 실행 가능하지만, 자동화된 백업이 없으므로 출시 전 최소 1회 수동 백업을 실행하고 cron을 설정해야 함.

**프로덕션 서버 필수 실행 명령**:
```bash
# 1. 수동 백업 즉시 실행 (DB 삭제 없이, 읽기 전용)
docker compose exec -T db pg_dump -U telegram_dashboard telegram_dashboard > backup_$(date +%Y%m%d_%H%M%S).sql

# 2. 백업 파일 무결성 확인 (파일 크기 > 0, SQL 문 포함)
ls -lh backup_*.sql
head -20 backup_*.sql

# 3. cron 등록 (매일 새벽 3시)
echo "0 3 * * * cd /path/to/telegram-dashboard-backend && docker compose exec -T db pg_dump -U telegram_dashboard telegram_dashboard > backups/backup_$(date +\%Y\%m\%d).sql" | crontab -
```
